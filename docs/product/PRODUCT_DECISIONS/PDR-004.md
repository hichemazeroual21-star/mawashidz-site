# PDR-004: Hub lazy loading (top four cards)

| | |
|--|--|
| **Status** | Accepted |
| **Date** | 2026-07-23 |
| **Owner** | Founder / Product |

## Title

Why lazy loading (top four cards first)

## Reason

Members may eventually have **many** Hub cards (weather, prices, actions, marketplace hooks, AI slot). Loading and fetching **all** providers on account open hurts **mobile performance** and time-to-value. Most users need the **highest-priority** context first—especially the hero **“Today in your wilaya”** composite.

## Alternatives considered

| Alternative | Why not chosen |
|-------------|----------------|
| **Load all cards on open** | Slow on 3G/4G; one failing API blocks perception of the whole Hub |
| **Single bundled API for entire Hub** | Large payloads; hard to cache per provider; fights offline-first |
| **No Hub until all data ready** | Blank workspace; violates offline and action-first principles |
| **Pagination to separate page** | Breaks “daily workspace” metaphor |

## Decision

- **Eager:** render and fetch only the **top four** cards by rank (include wilaya hero when wilaya is set).
- **Deferred:** remaining providers via **intersection observer** / scroll, **dynamic import** per provider.
- **Offline:** cached payloads with “last updated” stamp; never empty if cache exists.

## Consequences

- Provider implementations must be **isolated** (separate fetch, separate cache keys).
- Ranking and personalization (PDR-005, constitution) determine which four load first.
- Performance budget is a **product requirement**, not optional optimization.
