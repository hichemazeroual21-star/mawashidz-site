# PDR-002: Card Provider architecture

## Title

Why Card Provider architecture (Smart Hub)

## Reason

MawashiDZ will add many services over years (weather, alerts, insurance, equipment, AI, etc.). Rebuilding the account/home page for each service is expensive and inconsistent. A **modular Hub** where each capability is a **Card Provider** keeps the UI shell stable and makes new integrations a matter of **hours, not days**.

## Alternatives considered

| Alternative | Why not chosen |
|-------------|----------------|
| **Monolithic hub page** | Every feature touches the same file; merge conflicts and regressions |
| **One new full page per service** | Navigation sprawl; members do not get a single daily home |
| **Direct API calls from UI components** | Untestable; vendor lock-in; no offline/cache layer |
| **One database table per future product** | Schema explosion before usage is proven |

## Decision

- **Hub Engine** (`mdz-hub-core`) + **Card Provider** contract: data source, visibility, refresh, cache, priority.
- UI renders only **normalized** `card_type` + `payload` + `meta`.
- **No Card → SQL:** Provider → Service → RPC/API → Database (see constitution).
- Server registry: minimal `hub_cards` rows; founders can **enable/disable** providers without UI redeploy.

## Consequences

- New verticals register a provider (+ optional deep link for heavy flows).
- Composite cards (e.g. “Today in your wilaya”) are orchestrator providers, not six duplicate UIs.
- Marketplace modules must stay **compatible** with the same provider model (constitution).
