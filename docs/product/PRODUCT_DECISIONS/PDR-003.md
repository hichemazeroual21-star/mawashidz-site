# PDR-003: Platform event bus

| | |
|--|--|
| **Status** | Accepted |
| **Date** | 2026-07-23 |
| **Owner** | Founder / Product |

## Title

Why event bus (domain events)

## Reason

As listings, messages, approvals, Hub cards, notifications, analytics, and achievements grow, **direct coupling** between modules (A calls B calls C) becomes unmaintainable. A small **event bus** lets one domain action fan out to subscribers without each feature knowing every consumer.

## Alternatives considered

| Alternative | Why not chosen |
|-------------|----------------|
| **Direct imports / callbacks everywhere** | Spaghetti; hard to test; circular dependencies |
| **Single “god” orchestrator service** | Becomes a bottleneck; still knows all modules |
| **Only database triggers** | Opaque to client Hub; poor for UI refresh and analytics naming |
| **Defer entirely** | Pain hits when marketplace + workspace ship together |

## Decision

- **Domain operations emit versioned events** (e.g. `AnimalListed`, `PurchaseRequestCreated`).
- Subscribers update: Hub refresh hints, statistics, notification center, analytics, achievements (phased).
- Producers do **not** manipulate Hub DOM directly; consumers subscribe via bus (+ server outbox/RPC where needed).

## Consequences

- New features declare **event contracts** alongside APIs.
- Notification center and founder analytics can listen to the same events as the Hub.
- Implementation can start minimal in P1–P2; **principle is fixed** now to avoid rework.
