# PDR-005: Server-side workspace preferences

| | |
|--|--|
| **Status** | Accepted |
| **Date** | 2026-07-23 |
| **Owner** | Founder / Product |

## Title

Why server-side preferences (not localStorage-only)

## Reason

Card **hide/dismiss**, pin order, and workspace personalization must **follow the user across devices** (phone, second phone, desktop). `localStorage`-only preferences cause confusing splits, break analytics alignment, and fail when members clear browser data.

## Alternatives considered

| Alternative | Why not chosen |
|-------------|----------------|
| **localStorage only** | No cross-device; lost on clear; weak for trusted product feel |
| **Dedicated `hub_user_preferences` table on day one** | Premature schema before usage proof (Founder P0 minimal rule) |
| **No personalization until v2** | Poor UX; everyone sees identical grid longer than necessary |
| **Client-only rank heuristics** | Cannot power founder analytics or consistent support |

## Decision

- **First:** persist hub/workspace preferences in **`profiles.hub_preferences` (jsonb)** on the server, protected by RLS.
- **Later:** introduce a dedicated table only if jsonb becomes insufficient (volume, indexing, audit).
- **Engagement events** remain append-only server-side (`hub_engagement_events`) for analytics—not preference storage.

## Consequences

- P1 can ship hide/dismiss with jsonb; migrations stay small in P0.
- Personalization ranking (role, wilaya, season, activity) reads/writes server hints.
- Implementers must not rely on hidden UI alone; preferences sync after login.
